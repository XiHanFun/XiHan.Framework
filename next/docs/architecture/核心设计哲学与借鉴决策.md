# 核心设计哲学与借鉴决策

## 一、文档目的

本文档用于回答两个核心问题：

1. `XiHan.Framework.Next` 应该借鉴哪些成熟开源项目和系统的核心设计思想。
2. 在借鉴过程中，哪些只是“哲学层启发”，哪些可以真正落到工程结构上。

本文档不直接讨论具体代码实现细节，而是用于统一后续所有核心项目的设计方向，避免在架构演进中反复摇摆。

## 二、总原则

`XiHan.Framework.Next` 不复制任何单一框架或单一系统的完整模型。

我们采用的不是“照抄某个架构”，而是以下方法：

1. 吸收成熟生态中已经被长期验证的稳定原则
2. 放弃与 .NET 应用框架场景不匹配的过重机制
3. 把哲学性表达转换成可以落地的工程约束

最终形成的判断是：

> `XiHan.Framework.Next` 应采用“最小核心 + 分层引导 + 多连接机制并存 + 模块化可选”的路线，而不是“单一模块化中心”或“纯意图驱动内核”。

## 三、主要借鉴来源

### 1. Rust

Rust 最值得借鉴的不是语法，而是它对 `core / alloc / std` 的分层边界。

可借鉴点：

1. 最小核心必须极小
2. 更重能力放到上层
3. 核心一旦定义，就必须长期稳定

对 `XiHan.Framework.Next` 的映射：

1. `XiHan.Framework.Kernel` 只承载稳定原语
2. 引导、发现、生命周期编排进入 `Bootstrap`
3. 业务能力和 Provider 能力继续上移

### 2. Java / Spring / ServiceLoader

Java 生态中同时存在两类值得借鉴的设计：

1. `ServiceLoader` 代表的契约 + Provider + 可选发现
2. Spring Boot 代表的自动装配与条件装配

可借鉴点：

1. 抽象与实现分离
2. 模块或能力可以被发现，但发现机制不是唯一入口
3. 默认实现与抽象层要分离

对 `XiHan.Framework.Next` 的映射：

1. `Abstractions` 与具体实现项目分离
2. `Bootstrap.Abstractions` 与 `Bootstrap` 分离
3. 可以支持模块自动发现，但必须保留显式注册路径

### 3. Python

Python 生态更强调包、元数据和插件发现，而不是一个强中心化运行时。

可借鉴点：

1. 功能应具备自描述能力
2. 插件/扩展能力应建立在明确契约之上
3. 扩展机制应尽量降低对核心的侵入

对 `XiHan.Framework.Next` 的映射：

1. 模块能力应有明确元数据与契约
2. 后续插件系统应作为增强能力，而不是最小核心前提
3. 核心不应被动态扩展机制反向绑架

### 4. Linux / OpenHarmony

Linux 和 OpenHarmony 最值得借鉴的是“分层 + 子系统 + 组件协作”，不是“所有能力都塞进统一核心”。

可借鉴点：

1. 核心负责边界和规则，不负责一切实现
2. 子系统可以独立演进，但要遵守公共契约
3. 宿主装配和运行时能力是单独层次，不应污染最小核心

对 `XiHan.Framework.Next` 的映射：

1. `Kernel` 只保留最基础边界
2. `Bootstrap` 负责引导和运行时装配
3. `hosts` 负责最终运行入口

## 四、对两类参考文档的正式判断

### 1. 核心架构方案判断

以现代工程视角总结：

1. 内核 + 扩展
2. 模块化单体
3. 依赖注入
4. 生命周期钩子
5. 配置驱动
6. 事件总线
7. 插件系统
8. 中间件/管道

这些内容与 `XiHan.Framework.Next` 高度兼容，因此属于：

**可直接借鉴的工程设计参考。**

但也要控制边界：

1. 事件总线不能成为所有模块唯一协作方式
2. 插件系统不能在早期阶段做重
3. 生命周期钩子不能无限扩张

### 2. 哲学判断

定义：

1. 无为而治
2. 顺势而为
3. 意图驱动
4. 去中心化自化
5. 调和循环
6. 柔性容错

这些表达在哲学层面是有价值的，因为它提醒我们：

1. 核心不应过度控制模块细节
2. 运行时应更偏“纠偏”而不是“强指挥”
3. 架构需要柔性，而不是只追求刚性

但如果直接把这套模型落成通用应用框架核心，会有明显问题：

1. 抽象过度，成本高于收益
2. 对普通业务系统过重
3. 容易把应用框架做成“迷你编排平台”
4. 不适合作为当前 .NET 框架最小核心

因此正式判断是：

**可以借鉴其哲学启发，但不能直接借鉴其完整核心结构。**

## 五、XiHan.Framework.Next 的核心哲学

### 1. 内核要克制

内核存在的目的不是“装下最多功能”，而是“定义最稳定边界”。

因此：

1. 内核只放十年后仍然大概率成立的东西
2. 任何依赖宿主、扫描、应用工厂的能力都不进入内核
3. 内核不是杂物间

### 2. 装配要显式

现代框架最大的问题之一，是把太多行为藏进扫描、魔法注册和隐式约定里。

因此：

1. 显式注册优先
2. 自动发现可作为增强能力
3. 自动装配必须有明确边界与退出机制

### 3. 模块化不是信仰

模块化只是组织和运行时装配的一种方式，不是整个框架唯一合法形式。

因此：

1. 框架应支持纯组件化接入
2. 框架应支持抽象 + Provider 接入
3. 框架也支持模块化运行时接入
4. 但不强迫所有能力都必须写成模块

### 4. 柔性存在于边界内，刚性存在于边界上

这是本框架最重要的哲学之一。

含义是：

1. 在模块内部，允许实现灵活演化
2. 在运行时策略上，允许配置驱动和可选发现
3. 但在项目边界、依赖方向和分层规则上，必须刚性约束

换句话说：

1. 内部可以柔
2. 边界必须硬

### 5. 宿主负责最终落地

核心与引导层只提供能力与装配机制。

真正决定“怎么跑起来”的，是宿主。

因此：

1. `Kernel` 不感知具体宿主
2. `Bootstrap` 不反向依赖具体宿主
3. `hosts` 承担运行入口、WebApplication、协议适配和最终装配职责

## 六、推荐采用的核心连接方式

结合借鉴结论，`XiHan.Framework.Next` 推荐采用以下连接方式共存：

### 1. 依赖注入

作为最基础连接方式，负责对象图装配与能力组合。

### 2. 生命周期扩展点

作为引导层连接方式，用于服务注册后、宿主初始化前后的编排。

### 3. 配置驱动

作为功能启停、Provider 选择和运行策略切换的声明式入口。

### 4. 显式扩展方法

作为默认接入方式，通过 `AddXxx()`、`UseXxx()` 进行清晰组合。

### 5. 模块化运行时

作为中大型系统可选机制，用于：

1. 依赖声明
2. 生命周期编排
3. 集中启动装配

### 6. 事件机制

作为模块协作方式之一，用于解耦而不是代替所有调用。

### 7. 插件系统

作为后续增强方向保留，不纳入当前最小核心。

## 七、明确不采用的方向

为了避免后续反复偏移，以下方向明确不作为当前核心路线：

1. 不做“大一统 Core”
2. 不把模块化定义成唯一元架构
3. 不把反射扫描作为全框架默认入口
4. 不做通用 Reconcile Loop 式调和内核
5. 不做过重的插件平台式最小核心
6. 不用哲学隐喻替代工程边界

## 八、当前落地后的结构判断

结合当前已经完成的重构，现阶段结构判断如下：

1. `Kernel`
   已基本符合“最小核心”定位。
2. `Bootstrap.Abstractions`
   已基本符合“引导层抽象”定位。
3. `Bootstrap`
   已基本符合“默认引导与模块运行时实现”定位。
4. `Hosts.Api`
   已成为首个真实接入 `Bootstrap` 链路的宿主样板。

这说明当前架构路线与本文档的判断是一致的，不需要回退到旧 `Core` 路径，也不需要转向纯哲学化“意图驱动内核”。

## 九、后续决策建议

后续所有核心设计决策，建议按以下顺序判断：

1. 这个能力是否十年后仍应留在核心边界？
2. 它是稳定原语，还是运行时装配能力？
3. 它需要显式组合，还是需要可选发现？
4. 它是通用框架能力，还是宿主/模块/Provider 专属能力？
5. 它是否会让 `Kernel` 再次膨胀？

如果上述问题无法回答清楚，则说明该能力还不应进入核心项目。
