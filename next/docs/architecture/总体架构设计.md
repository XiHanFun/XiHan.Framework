# 总体架构设计

## 一、设计背景

现有 `framework/src` 已经具备一定分层意识，但在长期演进过程中暴露出几个典型问题：

1. 核心包逐渐膨胀，容易演化为全局依赖中心。
2. 工具类和基础能力边界模糊，导致职责扩散。
3. 抽象与实现的拆分方式不统一，扩展路径不稳定。
4. 模块、宿主、基础设施实现之间容易出现横向耦合。

`XiHan.Framework.Next` 的目标不是修补旧结构，而是建立一套从第一天起就强调稳定边界、可替换实现和长期治理的全新骨架。

同时，本工作区明确引入三条执行约束：

1. 依赖包优先使用微软官方包和社区高活跃成熟包。
2. 正式项目与测试项目在构建、打包、发布上严格隔离。
3. 目录结构尽量扁平，更多层次通过解决方案目录和项目命名表达。

## 二、总体目标

### 1. 架构目标

1. 内核稳定
2. 分层明确
3. 模块自治
4. 实现可替换
5. 宿主轻量
6. 测试可分层验证

### 2. 非目标

当前阶段不追求：

1. 一次性迁移所有历史能力
2. 与旧项目完全二进制兼容
3. 在没有约束的情况下快速复制旧代码

## 三、物理结构

本工作区采用“代码域少而清晰、项目名强表达、解决方案目录分层”的结构策略。

## `src/framework`

用于承载框架自身的通用构建块和基础设施实现。

`framework` 下直接采用扁平项目目录，不再额外引入 `building-blocks`、`providers` 这类中间目录。

当前推荐的项目组织方式例如：

- `src/framework/XiHan.Framework.Kernel`
- `src/framework/XiHan.Framework.Bootstrap.Abstractions`
- `src/framework/XiHan.Framework.Bootstrap`
- `src/framework/XiHan.Framework.Shared`
- `src/framework/XiHan.Framework.Domain.Primitives`
- `src/framework/XiHan.Framework.Domain`
- `src/framework/XiHan.Framework.Application.Contracts`
- `src/framework/XiHan.Framework.Application`
- `src/framework/XiHan.Framework.Data.Abstractions`
- `src/framework/XiHan.Framework.Data.EntityFrameworkCore`
- `src/framework/XiHan.Framework.Data.SqlSugar`
- `src/framework/XiHan.Framework.Logging.Abstractions`
- `src/framework/XiHan.Framework.Logging.Console`
- `src/framework/XiHan.Framework.Logging.Serilog`

即：

1. 物理目录扁平
2. 项目命名表达分层
3. 解决方案目录表达逻辑归类
4. 是否属于“抽象”或“具体实现”通过项目名体现，而不是通过额外目录层体现

其中框架底部采用三段式边界，而不是单一“大模块化核心”：

1. `Kernel`
   只承载最小核心原语、最小模块契约和稳定异常。
2. `Bootstrap.Abstractions`
   承载应用生命周期、引导上下文、可选模块运行时扩展点等抽象。
3. `Bootstrap`
   承载默认程序集发现、类型发现、后续应用工厂与运行时装配实现。

因此，模块化在本架构中是可选能力之一，而不是整套框架唯一的元模型。

## `src/modules`

用于承载业务模块。每个模块都采用统一结构：

1. `Domain`
2. `Application.Contracts`
3. `Application`
4. `Infrastructure`
5. `Web`

这样可以保证：

1. 模块内部边界稳定
2. 模块外部依赖可控
3. 模块能够独立演进
4. 模块具备被宿主按需组合的能力

模块目录同样优先采用“模块目录 + 扁平项目目录”的形式，而不是继续增加中间层。

同时，当前阶段不再预创建空模块目录，只有真正进入实现阶段的模块才创建对应项目。

## `src/hosts`

用于承载运行入口与协议适配层，如：

1. `XiHan.Framework.Hosts.Api`
2. `XiHan.Framework.Hosts.Gateway`
3. `XiHan.Framework.Hosts.Grpc`
4. `XiHan.Framework.Hosts.RealTime`

宿主层的职责是装配，而不是定义核心规则。

## 四、依赖规则

1. 宿主层可以依赖模块层和框架层。
2. 模块层可以依赖框架构建块和基础设施抽象。
3. 具体实现项目可以依赖相关抽象和必要的框架基础项目。
4. 抽象层不得依赖具体 Provider。
5. `Kernel` 不得承载宿主抽象、运行时发现机制和应用工厂。
6. `Kernel` 不得反向依赖上层模块或宿主。
7. 模块之间默认不得直接互相引用实现层，只允许通过契约、集成事件或共享抽象协作。

## 五、依赖包策略

### 1. 选择优先级

基础依赖按以下优先级选择：

1. 微软官方包
2. 微软生态成熟官方扩展
3. 社区长期维护、下载量高、版本演进稳定的成熟包

### 2. 选择原则

1. 优先稳定性，不盲目追逐新奇功能
2. 优先生态兼容性，避免与 ASP.NET Core、Microsoft.Extensions 体系割裂
3. 优先长期维护活跃度
4. 优先清晰许可证和健康社区

## 六、构建、打包、发布隔离

正式项目和测试项目虽然位于同一工作区，但生命周期必须完全区分。

### 1. 正式项目

正式项目包括：

1. 框架项目
2. 模块项目
3. 宿主项目

这些项目参与：

1. 正式构建
2. 正式打包
3. 正式发布

### 2. 测试项目

测试项目只参与：

1. 测试构建
2. 测试执行
3. 测试覆盖率分析
4. 架构约束验证

测试项目不得进入正式发布产物，不得参与正式 NuGet 包输出。

## 七、关键设计原则

### 1. 先抽象，后实现

所有具备替换可能的能力，优先定义 `Abstractions`，再落地具体实现。

### 2. 先边界，后功能

任何新能力在开发前，先确定其属于：

1. 框架基础项目
2. 引导层抽象或引导层实现
3. 具体实现项目
4. 业务模块
5. 宿主层

如果一项能力依赖启动流程、服务提供器、程序集扫描、模块装配或宿主上下文，则默认不进入 `Kernel`。

### 3. 先组合，后耦合

模块协作优先通过接口、契约、事件和配置完成，而不是直接引用对方内部实现。

### 4. 先治理，后扩展

集中构建配置、集中包版本、统一中文文档和统一目录命名，是后续规模化演进的基础。

## 八、当前实施策略

当前阶段按以下顺序推进：

1. 固化目录结构
2. 固化解决方案与构建治理
3. 创建核心项目
4. 收口早期遗留目录偏移
5. 固化依赖方向
6. 增补中文文档和模板
7. 再逐步迁移或重写能力实现
