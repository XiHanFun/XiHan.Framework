# XiHan Framework — 代码规范

> 版本：1.0.0-preview.1 | 状态：草案 | 最后更新：2026-05-09

---

## 一、文件与项目规范

### 1.1 文件编码

| 规范 | 值 |
|------|-----|
| 编码 | UTF-8 without BOM |
| 换行符 | LF（`\n`） |
| 文件末尾 | 一个空行 |

### 1.2 文件头

**旧框架每文件 13 行版权头在新框架中全部移除**，改为简洁的版权声明：

```csharp
// Copyright (c) XiHanFun Contributors. Licensed under the MIT License.
// See the LICENSE file in the project root for full license text.
```

规则：
- 所有 `.cs`、`.csproj`、`.props` 文件包含以上两行
- 生成的代码（`.g.cs`、`*.generated.cs`）可省略

### 1.3 文件组织（每个 .cs 文件）

```csharp
// 1. 版权声明（仅根目录 NOTICE 或 LICENSE 引用即可，非每个文件必填）
// 2. using 指令（置于命名空间外）─ 系统命名空间在前，第三方在后，项目在最后
// 3. namespace（文件作用域）
// 4. 类型定义

using System.Collections.Immutable;
using Microsoft.Extensions.DependencyInjection;
using XiHan.Kernel.Capability;

namespace XiHan.Kernel.Hosting;

public sealed class XiHanAppBuilder { }
```

规则：
- `using` 语句置顶，置于命名空间外部
- 系统命名空间 → 第三方命名空间 → 框架内部命名空间，空行分隔
- 使用文件作用域命名空间 (`namespace X;`)
- 全局 using 在项目级 `.globalusings.cs` 或 `.csproj` 中定义
- 不使用 `#region`

### 1.4 项目文件（.csproj）

```xml
<Project Sdk="Microsoft.NET.Sdk">
  <PropertyGroup>
    <PackageId>XiHan.Framework.Kernel</PackageId>
    <Description>XiHan Framework Kernel — 微内核。</Description>
    <PackageTags>XiHan;Framework;Kernel</PackageTags>
    <RootNamespace>XiHan.Framework.Kernel</RootNamespace>
  </PropertyGroup>
  <ItemGroup>
    <PackageReference Include="Microsoft.Extensions.DependencyInjection.Abstractions" />
  </ItemGroup>
</Project>
```

规则：
- 每个包必须有 `<Description>` 和 `<PackageTags>`
- 最小依赖：只有必要的 `PackageReference`
- 版本号由 `Directory.Build.props` 统一管理
- 不锁死依赖版本（用中央包管理 `Directory.Packages.props`）

## 二、命名规范

### 2.1 通用规则

| 元素 | 风格 | 示例 |
|------|------|------|
| 命名空间 | PascalCase，匹配文件夹结构 | `XiHan.Framework.Kernel.Pipeline` |
| 类/结构体 | PascalCase | `PipelineBuilder` |
| 接口 | PascalCase + `I` 前缀 | `IPipelineMiddleware` |
| 枚举类型 | PascalCase | `Stability` |
| 枚举值 | PascalCase | `Stability.Stable` |
| 方法 | PascalCase | `UsePipeline()` |
| 属性 | PascalCase | `IsEnabled` |
| 局部变量 | camelCase | `builder` |
| 参数 | camelCase | `serviceProvider` |
| 私有实例字段 | `_camelCase` | `_middlewares` |
| 私有静态字段 | `s_camelCase` | `s_defaultOptions` |
| 常量 | PascalCase | `DefaultTimeout` |
| 泛型类型参数 | `T` 或 `T` + 描述 | `T`, `TFeature`, `TContext` |

### 2.2 框架特有约定

| 元素 | 约定 | 示例 |
|------|------|------|
| 能力接口 | `IXiHan` + 名词 | `IXiHanFeature` |
| 管道中间件 | `I` + 动词 + `Middleware` | `IValidationMiddleware` |
| 端口接口 | `I` + 名词 | `IRepository`, `IEventBus` |
| 扩展方法 | 在 `Extensions` 子命名空间 | `XiHan.Framework.Kernel.Hosting.Extensions` |
| Abstractions 包 | `XiHan.{Domain}.Abstractions` | `XiHan.Framework.Security.Abstractions` |
| 源生成器 | `XiHan.{Domain}.Generators` | `XiHan.Framework.Data.Generators` |

### 2.3 禁止的命名

- 无意义后缀：`Manager`、`Helper`、`Util`、`Common`
- 匈牙利命名法：`strName`、`intCount`
- 缩写（除非业界通用）：`ID`、`API`、`DTO`、`URL` 允许，`Mgr`、`Cfg`、`Svc` 禁止
- 中文标识符（含拼音）：禁止
- 多余的前缀/后缀：禁止 `XiHan` 出现在类型名中（命名空间已标识）

## 三、C# 代码风格

### 3.1 类型定义

```csharp
// ✅ 正确：简洁、只实现需要的接口
public sealed class PipelineBuilder
{
    private readonly ImmutableList<Type>.Builder _middlewareTypes = ImmutableList.CreateBuilder<Type>();

    public PipelineBuilder Use<TMiddleware>() where TMiddleware : IPipelineMiddleware
    {
        _middlewareTypes.Add(typeof(TMiddleware));
        return this;
    }
}

// ❌ 错误：冗余注释、region、多余修饰符
#region 管道构建器
/// <summary>
/// 管道构建器类，用于构建执行管道
/// </summary>
public class PipelineBuilder  // 没有 sealed
{
    private ImmutableList<Type>.Builder _middlewareTypes; // 没有 readonly

    /// <summary>
    /// 使用指定类型的中间件
    /// </summary>
    /// <typeparam name="TMiddleware">中间件类型</typeparam>
    /// <returns>管道构建器</returns>
    public PipelineBuilder Use<TMiddleware>() where TMiddleware : IPipelineMiddleware { ... }
}
#endregion
```

### 3.2 接口定义

```csharp
// ✅ 正确：最小接口，单一职责
public interface IPipelineMiddleware
{
    Task InvokeAsync(PipelineContext context, PipelineHandler next);
}

// ✅ 带配置的能力接口
public interface IFeature
{
    string Name { get; }
    Stability Stability { get; }
    void Configure(IFeatureConfigurationContext context);
}

// ❌ 错误：大量方法在一个接口中
public interface IApplicationService
{
    Task<T> GetAsync<T>(Guid id);
    Task<IReadOnlyList<T>> GetListAsync<T>(QueryRequest request);
    Task<T> CreateAsync<T>(T dto);
    Task<T> UpdateAsync<T>(Guid id, T dto);
    Task DeleteAsync(Guid id);
    Task<long> CountAsync<T>(QueryRequest request);
}
```

### 3.3 方法

```csharp
// ✅ 正确：短方法、清晰意图
public XiHanApp Build()
{
    ConfigurePipeline(_pipelineBuilder);
    return new XiHanApp(_services.BuildServiceProvider(), _pipeline);
}

// ❌ 错误：长方法、多职责
public XiHanApp Build()
{
    // 50 行配置逻辑
    // 30 行验证逻辑
    // 20 行初始化
}
```

约束：
- 方法体不超过 30 行（源生成器生成的代码例外）
- 参数数量不超过 4 个（超过使用配置对象）
- 避免 `out` 和 `ref` 参数
- 优先返回具体类型（`PipelineBuilder`），接口作为方法参数（`IPipelineMiddleware`）

### 3.4 属性 vs 方法

```csharp
// 属性：简单的数据访问，无副作用
public string Name { get; }

// 方法：有计算、有副作用、有异步
public Task<Result> ExecuteAsync(Context context);
```

### 3.5 空值处理

```csharp
// ✅ 启用 Nullable Reference Types (NRT)
public string? OptionalValue { get; set; }     // 可为空
public string RequiredValue { get; set; } = ""; // 不可为空

// ✅ 参数验证用 ArgumentNullException.ThrowIfNull（.NET 6+）
public void Configure(IServiceCollection services)
{
    ArgumentNullException.ThrowIfNull(services);
}
```

### 3.6 Immutable 和 Records

```csharp
// ✅ 数据载体用 record
public sealed record Error(string Code, string Message);

// ✅ 配置用 record
public sealed record PipelineOptions
{
    public bool ThrowOnError { get; init; } = true;
    public TimeSpan Timeout { get; init; } = TimeSpan.FromSeconds(30);
}
```

## 四、注释与文档

### 4.1 何时写注释

| 情况 | 是否写注释 |
|------|-----------|
| 显而易见的代码 | 不写 |
| 非显而易见的 WHY | 写一行 |
| 公共 API | 写 XML 文档注释（`<summary>`） |
| 性能敏感的决策 | 写一行说明 |
| 对某个 bug 的 workaround | 必须写，标明 bug 编号 |
| 临时 hack | 写 `// HACK: reason` |

### 4.2 禁止的注释

```csharp
// ❌ 禁止：装饰性块注释
#region <<版权版本注释>>
// ----------------------------------------------------------------
// Copyright ©2021-Present ZhaiFanhua All Rights Reserved.
// ...
// ----------------------------------------------------------------
#endregion

// ❌ 禁止：改代码不删旧注释（保留注释掉的代码）
// var oldCode = DoSomething();

// ❌ 禁止：废话注释
// 设置名称
public string Name { get; set; }

// ❌ 禁止：署名 + 日期注释（Git 已记录）
// Author:zhaifanhua
// Email:me@zhaifanhua.com
// CreateTime:2024/10/26 19:43:06
```

### 4.3 XML 文档注释（公共 API 必须）

```csharp
/// <summary>
/// 定义管道中处理请求的中间件组件。
/// </summary>
public interface IPipelineMiddleware
{
    /// <summary>
    /// 执行中间件逻辑并调用管道中的下一个中间件。
    /// </summary>
    /// <param name="context">当前请求的管道上下文。</param>
    /// <param name="next">管道中下一个中间件的委托。</param>
    /// <returns>表示异步操作的 Task。</returns>
    Task InvokeAsync(PipelineContext context, PipelineHandler next);
}
```

## 五、异常处理

### 5.1 异常设计

```csharp
// 框架基础异常
public class XiHanException : Exception
{
    public string Code { get; }
    public XiHanException(string code, string message) : base(message) => Code = code;
}

// 具体异常
public class CapabilityResolutionException : XiHanException
{
    public string CapabilityName { get; }
    public CapabilityResolutionException(string capabilityName, string message)
        : base("CAP_RESOLVE_001", message)
        => CapabilityName = capabilityName;
}
```

规则：
- 所有框架异常继承 `XiHanException`
- 异常带错误码（`CAP_RESOLVE_001` 格式：`{域}_{类型}_{序号}`）
- 可恢复的错误不用异常，用 `Result<T>` 返回

### 5.2 Result 模式（可恢复错误）

```csharp
// 避免异常用于控制流
public readonly struct Result<T>
{
    public T? Value { get; }
    public Error? Error { get; }
    public bool IsSuccess { get; }
    public bool IsFailure => !IsSuccess;

    private Result(T value) { Value = value; IsSuccess = true; Error = null; }
    private Result(Error error) { Error = error; IsSuccess = false; Value = default; }

    public static Result<T> Success(T value) => new(value);
    public static Result<T> Failure(Error error) => new(error);
}
```

## 六、日志规范

### 6.1 日志级别使用

| 级别 | 场景 |
|------|------|
| `Trace` | 极详细信息，仅开发时启用 |
| `Debug` | 调试信息，发版时不输出 |
| `Information` | 关键状态变更：启动完成、租户创建 |
| `Warning` | 非预期但可恢复：重试、降级、权限不足 |
| `Error` | 需要关注的错误：异常、操作失败 |
| `Critical` | 系统级故障：数据库不可用 |

### 6.2 结构化日志

```csharp
// ✅ 正确：结构化参数
_logger.LogInformation("Feature {FeatureName} configured in {ElapsedMs}ms", name, elapsedMs);

// ❌ 错误：字符串插值（丢失结构化信息）
_logger.LogInformation($"Feature {name} configured in {elapsedMs}ms");
```

## 八、依赖管理规范

### 8.1 依赖方向

```
Kernel ← Capability ← Feature 包
   ↑
Abstractions（无任何项目依赖）
```

### 8.2 第三方依赖规则

| 规则 | 说明 |
|------|------|
| 内核零第三方依赖 | 仅 `Microsoft.Extensions.*` |
| 扩展包最小传递 | 只引入必需的第三方库 |
| 版本集中管理 | `Directory.Packages.props` |
| 安全审计 | 每个 PR 检查依赖漏洞 |

### 8.3 API 封装规则

扩展包封装第三方库时：
- 不直接暴露第三方类型在公共 API 中
- 通过接口包装，内部委托给第三方实现
- 例如：`IRedisCache` 不暴露 `StackExchange.Redis.IDatabase`

## 九、Git 提交规范

### 9.1 提交消息

```
<type>(<scope>): <subject>

类型：feat / fix / docs / style / refactor / perf / test / chore
范围：包含名（kernel / capability / hosting / pipeline / data）
标题：简短描述，英文，小写，不加句号

示例：
feat(kernel): add pipeline builder
fix(capability): resolve circular dependency detection
docs(philosophy): add daoist design principles
```

### 9.2 分支策略

```
main          ← 稳定发布
release/v1.x  ← 发布分支
feature/*     ← 功能分支
fix/*         ← 修复分支
```

### 9.3 PR 要求

- 关联一个 issue
- 通过 CI（构建 + 测试 + lint）
- 至少一人审查
- 公共 API 变更必须在 PR 描述中标注
