# 核心项目归属判定规则

## 一、文档目标

本文档用于解决一个高频问题：

> 当新增一个能力时，它到底应该进入 `Kernel`、`Bootstrap.Abstractions`、`Bootstrap`、`Application`、`Web`、具体 `Provider`，还是 `modules`？

如果这个问题没有固定判断规则，框架在演进中很容易再次出现以下问题：

1. `Kernel` 被不断塞入“暂时没地方放”的代码
2. `Bootstrap` 与 `Web` 边界混淆
3. 宿主代码反向渗透到框架层
4. 抽象层与实现层职责漂移

因此，所有新能力接入前，都应先通过本文档做归属判断。

## 二、总原则

### 1. 先判断边界，再写代码

任何新能力在开始实现前，都要先回答：

1. 这是稳定原语，还是运行时能力
2. 这是抽象，还是默认实现
3. 这是框架能力，还是业务能力
4. 这是通用能力，还是宿主专属能力

### 2. 默认优先放更高层，而不是更底层

如果一个能力既可以放底层，也可以放上层，默认优先放上层。

原因是：

1. 上层更容易调整
2. 底层一旦膨胀，很难再拆
3. 最小核心必须长期稳定

### 3. 不允许“因为方便”下沉到核心

以下理由都不能成为把代码放进 `Kernel` 的依据：

1. “很多地方都能用到”
2. “先放核心后面再拆”
3. “看起来像基础能力”
4. “别的项目引用它比较方便”

## 三、判定总流程

新增能力时，按以下顺序判断：

1. 它是否属于业务规则或业务模型？
2. 它是否依赖宿主、WebApplication、HTTP、ASP.NET Core 或协议适配？
3. 它是否依赖启动流程、生命周期、应用工厂、模块图、服务提供器装配？
4. 它是否是一个可替换技术能力的抽象？
5. 它是否只是某个抽象的具体实现？
6. 它是否是十年后仍然稳定的原语？

根据答案进入对应项目。

## 四、项目归属判断表

### 1. `XiHan.Framework.Kernel`

适合进入 `Kernel` 的能力：

1. 最小模块契约
2. 最基础异常类型
3. 极稳定的对象访问器
4. 极稳定的类型原语
5. 十年后大概率不变的最小基础约定

必须同时满足以下特征：

1. 不依赖宿主
2. 不依赖 ASP.NET Core
3. 不依赖启动流程
4. 不依赖服务提供器构建过程
5. 不依赖程序集发现或模块扫描
6. 不依赖具体基础设施

典型例子：

1. `IXiHanModule`
2. `XiHanModule`
3. `DependsOnAttribute`
4. `IObjectAccessor<T>`
5. `XiHanException`

典型反例：

1. 应用工厂
2. 模块管理器
3. 程序集查找器
4. 类型查找器
5. WebApplication 访问器

### 2. `XiHan.Framework.Bootstrap.Abstractions`

适合进入 `Bootstrap.Abstractions` 的能力：

1. 应用生命周期抽象
2. 模块加载抽象
3. 模块管理抽象
4. 模块描述抽象
5. 引导过程中的上下文对象
6. 引导层扩展点接口

判断特征：

1. 它不属于最小核心
2. 它与应用启动和运行时装配有关
3. 它是契约，不是默认实现
4. 它应该能被不同宿主或不同运行时重用

典型例子：

1. `IXiHanApplication`
2. `IModuleLoader`
3. `IModuleManager`
4. `ApplicationInitializationContext`

典型反例：

1. `XiHanApplicationFactory`
2. 默认模块依赖排序实现
3. 默认程序集扫描实现

### 3. `XiHan.Framework.Bootstrap`

适合进入 `Bootstrap` 的能力：

1. 默认应用工厂
2. 默认模块装配实现
3. 默认模块依赖排序
4. 默认生命周期编排
5. 默认程序集发现和类型发现
6. `IServiceCollection` 接入扩展

判断特征：

1. 它依赖引导层抽象
2. 它是运行时装配默认实现
3. 它不应下沉到 `Kernel`
4. 它又不属于某个具体宿主

典型例子：

1. `XiHanApplicationFactory`
2. `ModuleLoader`
3. `ModuleManager`
4. `ServiceCollectionApplicationExtensions`

典型反例：

1. `WebApplication` 路由注册
2. ASP.NET Core 中间件链
3. 控制器发现和端点映射

### 4. `XiHan.Framework.Application.Contracts`

适合进入 `Application.Contracts` 的能力：

1. DTO
2. 应用服务契约
3. 请求/响应契约
4. 面向外部使用者的应用层接口

判断特征：

1. 它描述应用层输入输出
2. 它不包含应用层默认实现
3. 它可以被 Web、Host 或客户端复用

### 5. `XiHan.Framework.Application`

适合进入 `Application` 的能力：

1. 应用服务默认实现
2. 应用层协调逻辑
3. DTO 与领域模型转换
4. 应用层校验、编排和事务边界协作

判断特征：

1. 它不是业务领域模型本身
2. 它不是宿主装配逻辑
3. 它是应用层实现，不是契约

### 6. `XiHan.Framework.Web`

适合进入 `Web` 的能力：

1. 通用 Web 装配入口
2. ASP.NET Core 扩展
3. 请求管道接入扩展
4. 与控制器、端点、中间件、HTTP 上下文相关的通用能力

判断特征：

1. 它明确依赖 ASP.NET Core
2. 它是通用 Web 能力，而不是某个具体宿主私有逻辑
3. 它可以被多个宿主复用

典型例子：

1. `UseXiHanFramework()`
2. 全局异常处理中间件
3. 统一模型绑定或通用端点装配

典型反例：

1. 某个宿主项目自己的首页路由
2. 某个网关专属反向代理规则

### 7. `XiHan.Framework.*.Abstractions`

适合进入各种 `Abstractions` 项目的能力：

1. 日志抽象
2. 本地化抽象
3. 多租户抽象
4. 校验抽象
5. 安全抽象
6. 数据访问抽象

判断特征：

1. 它是可替换技术能力的契约
2. 它不依赖具体实现包
3. 它可被多个 Provider 共享

### 8. `XiHan.Framework.*` 具体实现项目

适合进入具体实现项目的能力：

1. `Console` 日志实现
2. `Serilog` 日志实现
3. `EntityFrameworkCore` 数据实现
4. `SqlSugar` 数据实现

判断特征：

1. 它绑定具体技术栈
2. 它依赖某个外部包或 Provider
3. 它不应反向污染抽象层

### 9. `src/hosts/*`

适合进入宿主项目的能力：

1. 程序入口
2. `WebApplication` 构建
3. 宿主环境配置
4. 协议适配
5. 最终端点装配
6. 运行时进程生命周期处理

判断特征：

1. 它明确属于某个运行入口
2. 它无法被多个宿主复用
3. 它与最终运行形态强绑定

### 10. `src/modules/*`

适合进入业务模块的能力：

1. 业务领域模型
2. 业务应用服务
3. 业务基础设施实现
4. 业务 Web 接口

判断特征：

1. 它表达的是业务，而不是框架
2. 它不应进入 `src/framework`

## 五、快速判断问卷

如果需要快速判断一个能力的归属，按下面问题顺序回答：

1. 它是不是业务规则？
   是：进入 `modules`
2. 它是不是某个具体技术实现？
   是：进入对应实现项目
3. 它是不是一个可替换能力的契约？
   是：进入对应 `Abstractions`
4. 它是不是 ASP.NET Core 通用 Web 能力？
   是：进入 `Web`
5. 它是不是引导层契约？
   是：进入 `Bootstrap.Abstractions`
6. 它是不是引导层默认实现？
   是：进入 `Bootstrap`
7. 它是不是十年后仍然稳定的最小原语？
   是：进入 `Kernel`
8. 如果以上都不是：
   说明能力边界还不清晰，先不要实现，先补设计。

## 六、几个典型案例

### 案例 1：模块依赖声明特性

能力：`DependsOnAttribute`

判断：

1. 不是业务规则
2. 不是 Web 能力
3. 不是引导实现
4. 是最小模块契约的一部分

结论：

进入 `Kernel`

### 案例 2：应用工厂

能力：`XiHanApplicationFactory`

判断：

1. 与应用启动相关
2. 属于引导实现
3. 不属于最小核心

结论：

进入 `Bootstrap`

### 案例 3：WebApplication 路由装配扩展

能力：`UseXiHanFramework()`

判断：

1. 依赖 ASP.NET Core
2. 可被多个宿主复用
3. 不是某个宿主私有逻辑

结论：

进入 `Web`

### 案例 4：数据库连接字符串解析抽象

能力：连接字符串解析接口

判断：

1. 属于可替换数据访问能力契约
2. 不是最小核心

结论：

进入 `Data.Abstractions`

### 案例 5：订单聚合根

能力：`OrderAggregate`

判断：

1. 明确是业务领域模型

结论：

进入 `modules`

## 七、强制约束

以下红线默认禁止突破：

1. `Kernel` 不得引用宿主、Web、Provider 或 ASP.NET Core
2. `Bootstrap` 不得反向依赖具体宿主
3. `Web` 不得承载具体业务规则
4. `Abstractions` 不得引用具体实现项目
5. `modules` 不得把业务逻辑下沉到 `framework`

## 八、配套建议

为了让本规则真正生效，建议配套执行：

1. 继续增加架构测试
2. 新增能力前先补一段“归属说明”
3. PR 或改动审查时先审归属，再审实现
4. 不允许以“先放这里后面再拆”作为入库理由
