# ADR-0001 核心架构路线决策

## 状态

已采纳

## 日期

2026-04-21

## 背景

`XiHan.Framework` 旧架构长期存在以下问题：

1. 核心项目承担过多职责，逐步膨胀为全局依赖中心
2. 模块化机制被不断扩大，逐渐成为所有能力的唯一组织方式
3. 宿主、生命周期、扫描、装配、运行时控制逻辑混入核心
4. 抽象层与实现层拆分不稳定，扩展边界不清晰

在重构 `XiHan.Framework.Next` 时，需要明确回答以下问题：

1. 新框架是否仍然以模块化作为唯一元架构
2. 是否借鉴著名开源项目与系统的核心设计
3. 哪些设计思想只适合作为哲学启发，哪些可以真正落地到工程结构中

## 决策

`XiHan.Framework.Next` 采用以下核心架构路线：

1. 最小核心
   `Kernel` 只承载最稳定原语与最小模块契约。
2. 分层引导
   引导层抽象进入 `Bootstrap.Abstractions`，默认引导实现进入 `Bootstrap`。
3. 多连接机制并存
   使用 DI、生命周期扩展点、配置驱动、显式扩展方法、可选模块运行时、事件机制等多种连接方式。
4. 模块化可选
   模块化保留，但只作为中大型系统的可选运行时能力之一，不再作为全框架唯一组织方式。
5. 宿主最终装配
   `hosts` 负责运行入口与最终装配，核心与引导层不反向依赖具体宿主。

## 借鉴来源

### 1. Rust

借鉴 `core / alloc / std` 的分层方式：

1. 核心极小
2. 更重能力逐层上移
3. 最小核心长期稳定

### 2. Java / Spring / ServiceLoader

借鉴以下工程模式：

1. 契约与实现分离
2. Provider 可选发现
3. 自动装配与默认实现分层承载

### 3. Python

借鉴以下扩展思想：

1. 功能具备自描述能力
2. 插件建立在明确契约上
3. 扩展机制尽量不侵入核心

### 4. Linux / OpenHarmony

借鉴以下系统级结构原则：

1. 分层而不是堆叠
2. 子系统协作而不是单一核心吞并一切
3. 核心负责规则与边界，不负责所有实现

## 不采纳的方向

以下方向明确不作为当前核心路线：

1. 不做大一统 `Core`
2. 不把模块化定义成唯一元架构
3. 不把反射扫描定义成全框架默认唯一入口
4. 不构建通用 Reconcile Loop 式调和内核
5. 不在最小核心内引入过重插件平台机制
6. 不用哲学隐喻替代工程边界

## 哲学原则

### 1. 内核要克制

内核的作用是定义稳定边界，而不是堆积能力。

### 2. 装配要显式

显式注册优先，自动发现只能是增强能力，不能成为唯一入口。

### 3. 模块化不是信仰

模块化可以存在，但不能要求所有能力都必须写成模块。

### 4. 柔性在边界内，刚性在边界上

框架允许模块内部灵活演化，但项目边界、依赖方向和分层规则必须强约束。

### 5. 宿主负责最终落地

核心与引导层提供能力和装配机制，宿主决定系统如何真正运行。

## 后果

采用本决策后，带来的正向结果包括：

1. `Kernel` 的职责长期可控
2. 运行时装配能力有明确承载层
3. 抽象、实现、宿主的依赖方向更清晰
4. 框架既能支持模块化单体，也能支持显式组件组合
5. 后续引入插件、事件、Provider、模块运行时时不会再次污染最小核心

同时也意味着：

1. 早期项目数量会略多于“单一大核心”方案
2. 需要通过文档和架构测试持续维持边界
3. 某些功能接入时需要先判断归属层，而不能直接放入“核心”

## 当前落实情况

本决策已在当前结构中开始落实：

1. `XiHan.Framework.Kernel`
2. `XiHan.Framework.Bootstrap.Abstractions`
3. `XiHan.Framework.Bootstrap`
4. `XiHan.Framework.Hosts.Api`
5. 架构边界守护测试
