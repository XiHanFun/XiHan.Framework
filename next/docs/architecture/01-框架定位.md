# XiHan Framework — 框架定位

> 版本：1.0.0-preview.1 | 状态：草案 | 最后更新：2026-05-09

---

## 一、框架定义

### 1.1 一句话定位

**XiHan Framework 是面向 .NET 的企业级通用框架，以"微内核 + 能力图谱 + 管道执行"为核心范式，适用于 B2B SaaS、企业后台、微服务等多种场景。**

### 1.2 三层定位

| 层 | 定位 | 对标参照 |
|----|------|---------|
| **底层框架** | .NET 生态的企业级应用基础设施 | ASP.NET Core 原生能力的延伸和补充 |
| **中间能力** | 通用企业功能标准化（多租户、权限、数据隔离、缓存、消息等） | Dapr Building Blocks 的思路 |
| **上层工具链** | 开发效率工具（动态 API、代码生成等） | 按需启用，非强制绑定 |

### 1.3 主要适用场景

| 场景 | 特征 |
|------|------|
| **多租户 SaaS 平台** | 租户隔离、数据分区、订阅计划 |
| **企业后台管理系统** | 复杂权限、审批流、组织架构 |
| **行业垂直应用** | CRM/ERP/WMS/HRM 等行业定制 |
| **微服务后端** | 每个微服务独立使用框架核心能力 |
| **开放 API 平台** | 多端接入、速率限制、灰度路由 |
| **边缘/轻量服务** | 仅加载必需能力包，最小化运行时 |

### 1.4 不是

| 不是 | 原因 |
|------|------|
| 不是 ABP 的替代品或分支 | 架构哲学不同：特性组合 vs 模块化生命周期 |
| 不是 Spring Boot 的 .NET 移植 | .NET 有自己的最佳实践（HostBuilder、Source Generator） |
| 不是微服务框架 | 不绑部署形态，单体/模块化/微服务均可 |
| 不是低代码平台 | 面向开发者，不面向拖拽配置 |
| 不是 ORM | 不绑特定 ORM，通过端口模式适配 |
| 不是 UI 组件库 | 前后端分离，前端独立演进 |

### 1.5 是

| 是 | 具体含义 |
|-----|---------|
| 开发效率基础设施 | 提供企业级通用能力，减少重复编码 |
| 架构共识载体 | 为团队提供统一的架构约定、代码规范、分层标准 |
| 能力集成平台 | 通过能力图谱协调多技术栈，而非基类继承绑定 |
| 质量保障体系 | 内置可观测、审计、错误模型、API 稳定性分级 |
| 可扩展宿主 | 最小内核 + 可选扩展包，按需组合，渐进增长 |

## 二、与主流 .NET 框架对比

### 2.1 ABP Framework

| 维度 | ABP | XiHan |
|------|-----|-------|
| 模块化 | 强制模块系统，`[DependsOn]` 声明式依赖 + 6 个生命周期钩子 | 模块化为可选组织模式，`IXiHanFeature` + `UseXxx()` 显式组合，无强制生命周期 |
| 实体基类 | 强制继承 `Entity<TKey>` 等 | 仅要求实现 `IEntity<TKey>` 接口 |
| 扩展机制 | 拦截器 + 虚方法重写 | 管道中间件 + 源生成器 |
| 动态代理 | Castle Windsor 运行时代理 | 编译时源生成器 + 管道委托 |
| 包数量 | 50+ | V1 9 个包（8 内核 + 1 测试） |
| DI | 包装 MS DI + 自动注册 | 原生 MS DI + 显式注册 |
| 前端 | ASP.NET Zero / LeptonX | 前后端分离，仅提供 API 契约 |

### 2.2 Furion

| 维度 | Furion | XiHan |
|------|--------|-------|
| 设计理念 | 大而全的"全家桶"，开箱即用 | 微内核 + 可选能力包，按需加载 |
| 依赖注入 | 基于属性的自动扫描注册 | 显式注册 + 可选自动扫描 |
| 动态 API | 内置核心功能 | 独立扩展包 `XiHan.Framework.Web.DynamicApi`（V2） |
| 配置方式 | 约定为主，配置为辅 | 显式配置为主，约定可覆写 |
| 包结构 | 单一主包 + 少量扩展 | 分层抽象包 + 多实现适配包 |
| ORM 绑定 | Furion 自身不绑 ORM，但社区主流搭配 EF Core | 端口模式，EF Core / SqlSugar / Dapper 各有适配包 |
| 目标场景 | 中小型项目快速开发 | 中大型企业项目、长期维护项目 |

### 2.3 MASA Framework

| 维度 | MASA Framework | XiHan |
|------|---------------|-------|
| 设计理念 | 云原生、微服务优先 | 部署形态中立，单体/微服务均可 |
| 模块化 | Building Blocks 模式 | 能力图谱 + 管道模式 |
| 数据访问 | 多 ORM 支持（EF Core / Dapr State） | 端口抽象 + 适配包（EF Core / SqlSugar / Dapper） |
| 可观测性 | 集成 OpenTelemetry | 框架内置轻量实现，可桥接 OpenTelemetry |
| 多租户 | 内置 | 独立扩展包 `XiHan.Framework.Security.MultiTenancy` |
| 目标场景 | 云原生微服务 | 通用企业应用，不强制云原生 |

### 2.4 跨语言参照

| 参照 | 我们借鉴什么 | 我们不学什么 |
|------|------------|------------|
| **Dapr** | Building Blocks 能力抽象、端口-适配器模式 | Sidecar 架构（对大多数项目过重） |
| **NestJS** | 装饰器声明能力、模块化组织 | TypeScript 特有的装饰器元编程 |
| **Vue 3** | Composition API 的函数组合哲学 | 仅后端框架，不涉及 UI |
| **Rust** | 零成本抽象、编译时保证 | 语言层面的所有权机制 |

## 三、版本与兼容策略

### 3.1 TFM 滚动策略

```
.NET 10 (LTS, 支持至 2028-11)  ← 当前主力
.NET 9  (过渡期，12 个月内移除)
.NET 8  (LTS, 仅安全修复，24 个月内移除)
```

原则：**当前 LTS + 前一 LTS 过渡期**。不在单一 TFM 上锁定超过两年。

### 3.2 API 稳定性分级

| 级别 | 标记 | 含义 | 示例 |
|------|------|------|------|
| **Stable** | `[ApiLevel(Stability.Stable)]` | 承诺 3.x 主版本内不破坏 | `IXiHanApp` |
| **Preview** | `[ApiLevel(Stability.Preview)]` | 可能变更，不建议生产使用 | `ICapabilityNegotiator` |
| **Experimental** | `[ApiLevel(Stability.Experimental)]` | 可能移除，需反馈 | 新能力包的早期版本 |
| **Internal** | `[ApiLevel(Stability.Internal)]` | 不对外暴露，不承诺兼容 | `PipelineBuilder` |
| **Deprecated** | `[ApiLevel(Stability.Deprecated)]` | 将在一个主版本后移除 | 旧版 API |

### 3.3 弃用流程

```
V1.0: 标记 [Deprecated] + 编译警告 + 迁移文档
V1.1: 默认禁用 + 配置可启用
V2.0: 移除
```

### 3.4 版本号策略（SemVer 2.0）

```
MAJOR.MINOR.PATCH[-preview.N]

MAJOR: 破坏性 API 变更
MINOR: 新增向后兼容功能
PATCH: 向后兼容的 bug 修复
-preview.N: 预览版
```

## 四、2023-2026 旧框架诊断（新框架需求的来源）

### 4.1 旧框架架构概览

旧框架（`XiHan.Framework/framework/`）包含 **50 个 NuGet 包**，按 .slnx 分层如下：

```
1.Common:    Analyzers, Utils（2 包）
2.Metadata:  Metadata（1 包）
3.Core:      Core（1 包）— 模块化系统、DI、动态代理、AOP
4.Domain:    Domain.Shared, Domain（2 包）
5.Application: Application.Contracts, Application, MultiTenancy.Abstractions, MultiTenancy,
               Settings, Validation.Abstractions, Validation（7 包）
6.Infrastructure: AI, Authentication, Authorization, Bot, Caching, Castle, Data, DevTools,
                  DistributedIds, EventBus.Abstractions, EventBus, Http,
                  Localization.Abstractions, Localization, Logging, Messaging,
                  ObjectMapping, ObjectStorage, Observability, Script, SearchEngines,
                  Security, Serialization, Tasks, Templating, Threading, Timing,
                  Traffic, Uow, Upgrade, VirtualFileSystem（31 包）
7.Web:        Web.Api, Web.Core, Web.Docs, Web.Gateway, Web.Grpc, Web.RealTime（6 包）
```

### 4.2 旧框架的具体问题

#### 问题 1：模块生命周期过度设计

```csharp
// 旧代码：XiHanModule.cs — 6 个生命周期钩子
public abstract class XiHanModule :
    IPreConfigureServices,       // 配置前
    IXiHanModule,                // 配置中
    IPostConfigureServices,      // 配置后
    IOnPreApplicationInitialization,  // 应用初始化前
    IOnApplicationInitialization,     // 应用初始化
    IOnPostApplicationInitialization, // 应用初始化后
    IOnApplicationShutdown       // 应用关闭
```

**影响**：模块作者困惑于"该在哪个钩子写代码"，调试时调用栈深达 20+ 层。

#### 问题 2：`[DependsOn]` 隐式依赖图

```csharp
[DependsOn(typeof(XiHanAuthorizationModule))]
[DependsOn(typeof(XiHanMultiTenancyModule))]
public class XiHanWebApiModule : XiHanModule { }
```

**影响**：依赖图无法被 IDE 静态分析，循环依赖在运行时才报错，测试时必须加载完整模块树。

#### 问题 3：基类继承链过深（7 层）

```
EntityBase → CreationEntityBase → ModificationEntityBase → DeletionEntityBase
  → FullAuditedEntityBase → MultiTenantFullAuditedEntityBase → AggregateRootBase
    → MultiTenantAggregateRootBase
```

#### 问题 4：SqlSugar 硬绑定

Data 模块直接依赖 SqlSugar，仓储实现与 ORM 耦合，无法切换到 EF Core 或其他 ORM。

#### 问题 5：包粒度过细

50 个 NuGet 包对使用者造成"选择困难"，且包间版本升级需精确同步。

#### 问题 6：反射依赖过重

自动服务注册、动态 API、模块发现、Castle 动态代理均依赖运行时反射，影响 AOT 兼容和冷启动性能。

#### 问题 7：测试覆盖率低

10 个测试项目覆盖 50 个源码项目，大部分模块无单元测试。

#### 问题 8：不必要的基础设施模块

Bot、Script、Templating、Upgrade 等功能应在应用层而非框架层。

## 五、新框架设计目标

| 维度 | 旧框架 | 新框架目标 |
|------|--------|----------|
| 内核包数 | 50 | 9 个包（8 内核 + 1 测试） |
| 模块系统 | `[DependsOn]` + 6 个生命周期钩子 | `UseXxx()` Builder 显式组合 |
| 扩展机制 | Castle 动态代理 | 管道中间件 + 源生成器 |
| 实体模型 | 7 层基类继承 | 接口协议 + 源生成器 |
| 数据访问 | SqlSugar 绑定 | 端口模式，适配任意 ORM |
| API 稳定性 | 无分级 | Stable/Preview/Experimental/Internal/Deprecated |
| 可观测性 | 日志模块独立 | 框架内置轻量实现，可桥接 OpenTelemetry |
| 测试基础设施 | 基本无 | Testing 包提供测试基座 |
| 文档 | 无系统性文档 | 核心文档 + 每个包的 README |
| AOT 兼容 | 不支持 | 目标兼容（源生成器减少反射） |

## 六、V1 不做清单

| 功能 | V1 状态 | 说明 |
|------|---------|------|
| Bot / Script / Templating / Upgrade | 移除 | 应用层需求，非框架内核 |
| 后台管理 UI | 永不做 | 前后端分离 |
| 工作流引擎 / 全文搜索 / AI 集成 | V2+ | 扩展包 |
| gRPC / WebSocket / 实时通信 | V2+ | 扩展包 |
| 动态 API | **V2 扩展包** | 保留但不进 V1 内核，作为 `XiHan.Framework.Web.DynamicApi` 发布 |
| Castle 动态代理 | 移除 | 源生成器替代 |

## 七、与 BasicApp 的关系

`XiHan.BasicApp` 是基于旧框架构建的参考应用。新框架稳定后，BasicApp 将作为官方参考实现重建：

- **当前 BasicApp 模块**：CodeGeneration、Saas（多租户 + 权限 + 组织）
- **新 BasicApp 定位**：验证框架能力包的可用性，作为新用户的起点模板
