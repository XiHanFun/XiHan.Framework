# 内核与引导分层说明

## 一、为什么不把“模块化”定为唯一架构

在过去一代框架设计里，常见做法是把模块化、生命周期、程序集扫描、应用工厂、服务注册编排都放进同一个“核心”项目。

这种做法在中小规模阶段推进很快，但一旦框架开始承载多宿主、多 Provider、多业务模块和多种运行方式，就会出现三个问题：

1. 核心项目不断变厚，最终演化为全局依赖中心。
2. 模块化从一种组织手段，变成所有能力必须遵守的唯一入口。
3. 启动、扫描、装配、宿主差异都被绑定进核心，导致后续难以替换。

新一代结构不再把“模块化”视为唯一元架构，而是把它降级为可选运行时能力。

## 二、参考原则

新边界设计主要借鉴以下成熟生态的共同点：

1. Rust
   `core` 极小、依赖极少，更重能力交给上层 crate。
2. Java
   `ServiceLoader` 强调契约与 Provider 分离，发现机制是可选能力，而不是核心唯一入口。
3. Python
   更偏向“包 + 元数据 + 插件发现”，而不是大一统模块运行时。
4. Linux / OpenHarmony
   更强调分层、子系统、组件协作和宿主装配，而不是把一切压入单一核心。

这些系统和生态的共同点不是“完全不用模块化”，而是：

1. 核心极小
2. 契约先行
3. 实现后置
4. 发现机制可选
5. 宿主负责最终装配

## 三、XiHan.Framework.Next 的三段式边界

### 1. Kernel

`Kernel` 只承载：

1. 最小模块契约
2. 稳定原语
3. 基础异常
4. 极少量跨项目稳定约定

`Kernel` 明确不承载：

1. 应用工厂
2. 生命周期编排
3. 程序集扫描
4. 类型发现
5. 宿主上下文

### 2. Bootstrap.Abstractions

`Bootstrap.Abstractions` 负责承接：

1. 应用生命周期抽象
2. 应用初始化与关闭上下文
3. 可选模块生命周期扩展点
4. 程序集发现与类型发现抽象

这一层的作用，是把“引导层抽象”从最小核心中隔离出来。

### 3. Bootstrap

`Bootstrap` 负责承接：

1. 默认程序集发现实现
2. 默认类型发现实现
3. 后续应用工厂
4. 后续模块描述模型
5. 后续模块图构建与装配

这一层可以继续演进，但不会再污染最小核心。

## 四、模块化在新架构中的定位

模块化仍然保留，但定位发生变化：

1. 它是能力组织方式之一
2. 它是中大型系统的可选运行时装配方式
3. 它不是所有项目都必须使用的唯一模型

因此，未来框架应同时支持：

1. 纯组件化接入
   通过 `AddXxx()`、`UseXxx()` 等扩展方法显式组合。
2. 契约 + Provider 接入
   通过抽象层和实现层组合。
3. 模块化运行时接入
   在需要模块依赖声明、生命周期编排、集中装配时再启用。

## 五、后续实施原则

后续新能力接入时，按以下顺序判断归属：

1. 如果是稳定原语，进入 `Kernel`
2. 如果依赖启动流程或生命周期，进入 `Bootstrap.Abstractions`
3. 如果是默认发现或装配实现，进入 `Bootstrap`
4. 如果是具体技术实现，进入对应 `Abstractions / Provider`
5. 如果是业务能力，进入 `modules`

只有这样，才能避免重新回到旧 `Core` 的膨胀路径。
