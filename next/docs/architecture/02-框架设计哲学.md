# XiHan Framework — 框架设计哲学

> 版本：1.0.0-preview.1 | 状态：草案 | 最后更新：2026-05-09

---

## 一、三条根原则

```
1. 内核最小化 — 只定秩序，不做约束
2. 协议优于继承 — 接口定义边界，不强制基类层级
3. 显式优于隐式 — 代码即文档，组合即配置
```

这三条根原则派生所有具体设计决策。

### 1.1 设计灵感来源

Unix 的"小工具组合"思想、Rust 的零成本抽象理念、Vue 的渐进式设计，共同塑造了框架的哲学基调——内核只定义抽象的秩序，扩展包提供可替换的实现，内核不替业务做决定。

## 二、核心设计决策

### 2.1 抽象与实现解耦

```
抽象 → 内核接口、契约（XiHan.Framework.Kernel）
实现 → 扩展包（XiHan.Framework.Data.EntityFramework、XiHan.Framework.Caching.Redis）
```

内核不包含任何具体技术实现。`IRepository<T>` 是抽象，`EfCoreRepository<T>` 是实现。换 ORM 不需要改内核。每个能力域拆为抽象包和实现包，避免反向依赖。

### 2.2 模块化为可选模式

模块化不是被抛弃，而是从强制机制降级为可选组织模式。任何实现 `IXiHanFeature` 的类都可以作为一个"模块"来组织相关服务，通过 `UseFeature<T>()` 显式加入。不强制、不扫描、不注入生命周期钩子——用户决定是否模块化、如何模块化。

### 2.3 内核不干预

内核做减法，不替业务做决定：
- 不自动注册服务（用户显式注册）
- 不自动生成 Controller（动态 API 作为可选扩展包）
- 不自动推断约定（约定可覆写，非强制）
- 不隐藏执行细节（管道显式组装）

### 2.4 显式组合

```csharp
// 新框架 — 显式组合（编译时验证）
var app = XiHanApp.CreateBuilder(args)
    .UseFeature<AuthorizationFeature>()
    .UseFeature<MultiTenancyFeature>()
    .UsePipeline(pipeline => pipeline
        .Use<TenantResolveMiddleware>()
        .Use<AuthorizationMiddleware>())
    .Build();

// 旧框架 — 隐式声明（运行时解析）
[DependsOn(typeof(XiHanAuthorizationModule))]
[DependsOn(typeof(XiHanMultiTenancyModule))]
public class MyModule : XiHanModule { }
```

### 2.5 管道执行

每个请求通过一个显式排列的中间件管道执行。管道是白盒组合——可看到执行顺序，可插入/移除/重排，测试只需 mock 下一个委托。

```csharp
pipeline.Use<UnitOfWorkMiddleware>()
        .Use<ValidationMiddleware>()
        .Use<AuthorizationMiddleware>()
        .Use<CachingMiddleware>();
```

### 2.6 版本稳定

公共 API 是框架的"用户态 ABI"——一旦发布，不频繁翻动。破坏性变更走两阶段弃用流程。MAJOR 版本升级提供自动化迁移工具。公共 API 标记 Stability 级别。内部实现可随时重构，不承诺兼容。

### 2.7 扩展隔离

区分两类扩展，防止高风险代码影响全局：

- **核心扩展**（可信）：框架官方维护，与内核版本锁定，严格测试。
- **第三方插件**（不可信）：仅通过注册接口通信，不允许直接引用彼此类型。失败降级而不崩溃。

### 2.8 AOT 优先

框架从第一天起为 AOT 设计，不假设运行时可用：

- 内核零运行时反射。服务注册用源生成器替代程序集扫描。
- 动态代理用编译时管道委托替代 Castle 运行时代理。
- 插件系统分两层：**静态插件**在编译时已知，AOT 兼容；**动态插件**运行时加载，仅非 AOT 场景可用。
- 所有公共 API 标注 AOT 兼容性（`[RequiresDynamicCode]` 明确标记非 AOT 路径）。

### 2.9 渐进演进

V1 只做正确的事，不做完美的事。每个能力包独立版本、独立演进。实验性功能明确标记 Experimental。

## 三、架构原则

### 3.1 单一职责 — 包级

| 原则 | 规则 |
|------|------|
| 每个 NuGet 包只提供一个能力域 | `XiHan.Framework.Caching.Redis` 只做 Redis 缓存 |
| 抽象包与实现包分离 | `XiHan.Framework.Caching.Abstractions` 独立于 `XiHan.Framework.Caching.Redis` |
| 内核包零第三方依赖（除 `Microsoft.Extensions.*`） | 零传递依赖 |
| 扩展包最小依赖传递 | 只引入必需的第三方库 |

### 3.2 开闭原则 — 管道级

```csharp
// 对扩展开放：新中间件插入管道任意位置
// 对修改关闭：管道核心逻辑不因添加中间件而修改
```

### 3.3 依赖倒置 — 端口级

```
高层（Application） → 定义端口（IRepository）
低层（Infrastructure） → 实现端口（EfCoreRepository）
```

编译依赖指向高层，运行时依赖指向低层。

### 3.4 接口隔离 — 协议级

```csharp
// 按能力拆分的协议，而非大而全的基类
public interface IEntity<TKey> { TKey Id { get; set; } }
public interface ISoftDeletable { bool IsDeleted { get; set; } }
public interface IMultiTenant { Guid TenantId { get; set; } }
public interface IAuditable { DateTime CreatedAt { get; set; } }

// 实体按需组合：只实现需要的能力
public class Order : IEntity<Guid>, IMultiTenant, ISoftDeletable { }
```

### 3.5 最少知识 — 包间通信

```
包间只通过接口通信，不直接依赖具体类型。
A 包 → B 包的接口（in B.Abstractions）
A 包 ⊥ B 包的实现（不引用）
```

## 四、可见性控制

遵循 Rust 的可见性模型：默认 `internal`，显式 `public` 才对外承诺兼容。

```
内核包 → 所有内部类型默认 internal
       → 只有显式标记 public 的接口/类型属于公开 API
       → 公开 API 受 Stability 分级约束
       → 内部实现可随时重构而不破坏上层模块
```

每个包的 `InternalsVisibleTo` 仅对测试项目开放，不对其他包开放。

## 五、设计模式选择

| 模式 | 使用场景 | 不用场景 |
|------|---------|---------|
| **Builder** | XiHanApp 构建、管道构建 | 简单创建 |
| **Pipeline/Middleware** | 请求执行链 | 非请求处理场景 |
| **Ports & Adapters** | 数据、缓存、消息等基础设施 | 纯业务逻辑 |
| **Feature/Plugin** | 可选能力的装配 | 核心必须能力 |
| **Options** | 能力配置 | 运行时可变的设置 |
| **Source Generator** | 实体映射、仓储生成、过滤 | 业务逻辑生成 |

## 六、反模式清单

| 反模式 | 旧框架表现 | 新框架避免方式 |
|--------|----------|-------------|
| **上帝基类** | `XiHanModule` 含 6 个生命周期 + 配置方法 | 无基类，只实现接口 |
| **魔鬼属性** | `ServiceConfigurationContext` 在不正确时机访问抛异常 | Builder 模式确保编译时安全 |
| **隐式依赖** | `[DependsOn]` Attribute | 构建时显式声明，编译器验证 |
| **深层继承** | 7 层 Entity 基类 | 接口组合，0 层继承 |
| **反射魔法** | 自动注册、动态 API、Castle 代理 | 源生成器 + 显式注册 |
| **包爆炸** | 50 个包各自独立版本 | 抽象包 + 实现包分层 |
