# 文档与注释规范

本文档用于约束 `XiHan.Framework.Next` 工作区内所有文档、说明、README、XML 注释、代码注释和设计决策记录的书写方式。

## 一、语言要求

1. 默认全部使用中文。
2. 对外部技术名词、框架名、协议名、类名、接口名可以保留英文原文。
3. 中文说明应优先表达“为什么这样设计”和“这个组件负责什么”，而不是只描述“做了什么”。

## 二、适用范围

本规范适用于以下内容：

1. 根目录和各项目 `README.md`
2. `docs` 目录下的架构文档、指南文档、ADR 文档
3. C# XML 注释
4. 关键代码块的行内注释或块注释
5. 生成模板、脚本说明、测试说明

## 三、文档要求

### 1. README 要求

每个项目至少应包含以下内容：

1. 项目定位
2. 责任边界
3. 依赖方向
4. 不负责的内容
5. 典型使用方式

### 2. 架构文档要求

架构文档应重点说明：

1. 背景问题
2. 目标和非目标
3. 分层方式
4. 依赖规则
5. 扩展方式
6. 风险与约束

### 3. ADR 要求

每一条重要架构决策应记录：

1. 决策背景
2. 备选方案
3. 最终决策
4. 决策理由
5. 对系统的影响

## 四、代码注释要求

### 1. XML 注释要求

公共类型、接口、公开方法、关键属性默认应补充 XML 注释，说明：

1. 该成员的职责
2. 适用场景
3. 关键约束
4. 参数和返回值语义

### 2. 行内注释要求

仅在以下场景使用普通代码注释：

1. 设计意图不直观
2. 存在边界条件
3. 有特殊性能或一致性要求
4. 与外部系统或协议存在隐式约束

不允许写低价值注释，例如：

1. “给变量赋值”
2. “循环集合”
3. “返回结果”

### 3. 注释风格要求

1. 先讲原因，再讲动作。
2. 注释必须简洁、具体、可验证。
3. 注释不得与代码事实冲突。
4. 代码变更时必须同步更新注释。

## 五、命名与说明风格

1. 文档标题使用中文。
2. 目录名和项目名保持英文命名，说明文字使用中文。
3. 统一使用“职责边界”“依赖方向”“扩展点”“实现约束”等术语。
4. 避免使用模糊表达，如“可能”“应该差不多”“暂时这样”。

## 六、执行规则

从本工作区当前阶段开始，所有新增文档和注释默认遵守本规范。

如需引入英文原始文档，应补充中文摘要，不允许只保留纯英文说明而无中文上下文。

## 七、依赖与目录补充约束

### 1. 依赖包选择

新增依赖必须优先按以下顺序评估：

1. 微软官方包
2. .NET 基金会或成熟官方生态包
3. 社区中维护活跃、下载量高、长期稳定的成熟包

不得为了“功能全”随意引入小众、低活跃、维护状态不明的依赖。

### 2. 正式项目与测试项目隔离

文档和项目说明中必须明确区分：

1. 哪些项目属于正式交付产物
2. 哪些项目只用于测试
3. 哪些命令用于构建正式项目
4. 哪些命令用于执行测试

### 3. 目录深度控制

物理目录应尽量扁平，不为了追求“目录上很分层”而增加过多嵌套。

原则上：

1. 一级目录表达代码域
2. 二级目录表达逻辑分组或模块名
3. 具体层次更多依赖项目命名和解决方案目录表达
